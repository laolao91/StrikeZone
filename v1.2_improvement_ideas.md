# StrikeZone v1.2 — Code Review & Improvement Ideas

**Scope:** Full review of `StrikeZone_v1.0` (the active source tree — `StrikeZone_v1.1` contains only a packaged `.ehpk`, no source). Analysis only — nothing in this document has been implemented.

**Method:** Read every source/test/doc file, cross-referenced the three historical design documents (`docs/2026-05-08-strikezone-design.md`, `docs/superpowers/specs/2026-05-23-zone-image-rendering-design.md`, `docs/superpowers/plans/2026-05-23-zone-image-rendering.md`) and `HANDOFF.md` against the current code and git history to separate "intentional design" from "drift/regression." A second independent pass (Opus-tier) verified the two highest-stakes findings against the actual source before they're reported below.

---

## TL;DR — what to do first

1. **Fix the all-cascade-failure data loss in `main.ts` (P0).** Real pitch data is currently discarded and replaced with raw debug text.
2. **Decide the zone-display strategy deliberately, not by default.** The image cascade has never been confirmed to work on real hardware. Recommendation: make a redesigned text/grid zone the shipping default; keep the image cascade as an opt-in experiment.
3. **Clean up ~150 lines of dead code** (3 unused render functions, a stale comment) and **one stray network permission** (`react.dev`).
4. **Reconcile the docs** — README, HANDOFF, and the 2026-05-23 spec each describe a different version of the zone feature; none matches current code exactly.

## 1. Critical — P0

### 1.1 All-cascade-failure path destroys real pitch data, shows only debug text

**Where:** `src/main.ts:156-160`, inside `renderPitch()`

```ts
if (cascadeAllFailed) {
  await upgradeText(INFO_ID,   INFO_NAME,   formatZoneDiagnostic(cascadeAttempts, true))
  await upgradeText(SPLITS_ID, SPLITS_NAME, formatZoneDiagnostic(cascadeAttempts, true))
  return
}
```

**Problem:** `cascadeAllFailed` and `cascadeProbed` are session-level latches (`main.ts:67-68`) set once on the first pitch and never reset. If the image cascade fails completely on a user's device, **every subsequent pitch for the rest of the session** shows the literal SDK diagnostic string (e.g. `ZONE-DBG\nA:120x144 1b\n 3164chr sendFailed\n...\nALL FAILED`) in *both* the INFO and SPLITS columns — not pitch type, not velocity, not count, not splits. The header (score/inning/count) still updates, so the app looks alive while displaying nothing useful. This is the worst possible failure mode precisely because it's silent.

**Why it happened:** The original approved plan (`docs/superpowers/plans/2026-05-23-zone-image-rendering.md`, Step 4.5) specifies different, correct behavior:
```ts
const label = renderZoneText(pX, pZ, szTop, szBot).split('\n')[0]
await upgradeText(INFO_ID, INFO_NAME, `${label}\n${info}`)
await upgradeText(SPLITS_ID, SPLITS_NAME, formatZoneDiagnostic(cascadeAttempts, true))
```
Git history shows a later commit, `704cdd2 fix: show cascade diagnostic in INFO column (wider, no scroll needed)`, which appears to be a debugging change (to make the diagnostic more readable while bringing the cascade up on hardware) that was never reverted before being committed.

**Fix:** Restore the plan's behavior at minimum — `info` must always reach the user. Better: skip the raw `ZONE-DBG` text in production entirely (see §3.3) and show a clean location label instead.

**Severity:** Critical. This is the failure path for the one subsystem (`zone.ts`/the image cascade) with the most documented historical instability and *zero* confirmed hardware success (see §2.1) — i.e., the path most likely to actually be hit by a real user.

## 2. The zone-display strategy — the central open decision

### 2.1 The image cascade has never been confirmed to work on real hardware

**Where:** `src/glasses/zone.ts` (`CASCADE_CONFIG`, `renderZoneImage`, `nextCascadeStep`, `encodePNG1bit`/`encodePNG4bit`), `src/main.ts` (`runCascade`, `renderPitch`)

**Background:** The zone feature has gone through three design iterations:
1. **2026-05-08 (original spec):** inline ASCII grid in the INFO text column, no image container.
2. **Pre-2026-05-13:** an actual PNG-image attempt was built and abandoned after exhausting essentially every encoding approach available (`HANDOFF.md`'s "Image zone history" table: `canvas.toDataURL()` corrupt on iOS WKWebView, raw pixel arrays rejected, `CompressionStream` hangs WKWebView, base64 exceeds BLE limits at multiple sizes). HANDOFF.md's explicit conclusion: *"BLE image transfer is not viable for the zone on this device/SDK. Text-only approach is the right path forward."* The team shipped a standalone ASCII grid (commit `a46b392`), which user feedback called "not helpful in understanding the location of the pitch" (cells too tiny, no orientation reference).
3. **2026-05-23 (current spec):** a deliberate decision to re-attempt images anyway — *"Text alternatives are not acceptable — the glanceability of a spatial grid with a dot is the point."* This produced the current 4-step size/format cascade (120×144 1-bit → 80×96 1-bit → 40×48 4-bit → 40×48 1-bit), each step a hand-rolled pure-JS PNG encoder (manual CRC32, Adler32, uncompressed-DEFLATE stored blocks — no library) sent as a raw byte array via `updateImageRawData` (working around a separately-broken base64-string SDK path).

**The problem:** there is no later handoff or note confirming the cascade actually succeeds on hardware, or which step it typically lands on. The override of the 2026-05-13 "not viable" conclusion was made on design conviction, not new evidence — and the silence since suggests it may not have been validated. Meanwhile the cascade machinery is now substantial: 4 custom encoders, a BLE-error-driven state machine, a `imageSizeInvalid` rebuild-and-retry special case, and (per §1.1) a failure path that actively breaks the rest of the app.

**Legibility math on the worst-case fallback:** `drawZonePixels` (`zone.ts:277`) uses a 4px margin, so at the smallest step (40×48) the usable zone is ~32×40px holding a 3×3 grid with 2px-thick borders, and the dot radius floor (`Math.max(2, round(min(w,h)*0.05))`) is just 2px. That's a ~10px grid cell with a 4px dot competing with gridlines on a 4-bit greyscale display. It is very unlikely to be more glanceable than a well-designed text label — which undermines the *reason* images were chosen over text in the first place.

**Recommendation: make a redesigned text-based zone the v1.2 default; keep the image cascade as an explicit opt-in/experimental path, not the thing every user hits first.**

This is not a return to the rejected 2026-05-13 ASCII grid — that design's flaws (tiny cells, no orientation, unclear out-of-zone indicators) were real and shouldn't be repeated. But `renderZoneText()` (`zone.ts:212-240`) already exists, is fully unit-tested, and computes real 3×3 banding via `getDotPosition()` — it just needs a genuine redesign pass (see §4.1) rather than being resurrected as-is. Concretely for v1.2:
- Wire a redesigned text zone into the normal `renderPitch` path as the default.
- Gate the image cascade behind a flag (e.g. `?zoneImage=1`, mirroring the existing `?devReplay` pattern in `main.ts:390`) until someone can validate it end-to-end on real glasses and confirm which step it lands on.
- This also resolves §1.1 by construction — there's no "all cascade failed, nothing to show" state if text is the default.

Each image frame also costs ~0.5–2s over BLE with no compression — a real latency cost for live, fast-arriving pitch data, independent of whether it ever renders correctly.

### 2.2 `renderZoneText` doesn't represent out-of-zone location visually

**Where:** `src/glasses/zone.ts:212-240`

**Problem:** For out-of-zone pitches, `renderZoneText` renders an *empty* 3×3 grid — the only location signal is the compact label (`↑/R`, `↓/L`, etc.). Since a large fraction of real pitches are balls (out of zone), this is a meaningful gap in whichever path is used. A future-friendly redesign should give out-of-zone pitches a visible marker too (e.g. an arrow glyph just outside the relevant grid edge — the platform's `▲△▶▷▼▽◀◁` set is built for exactly this, per the design guidelines), not just text.

### 2.3 Stale zone image lingers behind non-pitch screens

**Where:** `src/main.ts:136-142`, `renderStandard()`

**Problem:** The comment at line 139 (`// ZONE is an image container — leave it showing last pitch or blank`) confirms this is intentional, but it means the last pitch's strike-zone art stays visible behind the "Select a game," loading, and error screens. Minor polish issue, worth a deliberate decision either way (e.g. clear it to blank via `renderZoneBlank` when leaving pitch-view) rather than leaving it as an accidental side effect of the container model.

## 3. UI/UX findings

### 3.1 `CHARS_PER_LINE` is a fixed constant; the INFO column width is not

**Where:** `src/glasses/display.ts:4` (`const CHARS_PER_LINE = 38`), used by `center()` (`display.ts:6-9`), which backs `renderHeader`, `renderGameList`, and `renderStateScreen` — all rendered into `INFO_ID` via `renderStandard`.

**Problem:** `38` matches the INFO column's width only when the cascade is on step A (120px zone image → 226px INFO column, per `zoneGeometry()` in `main.ts:72-78`, and matching the original design doc's own stated assumption of "~38 chars per line" for a 226px column). On steps B/C/D, the zone image shrinks (80px/40px) and `zoneGeometry` widens INFO correspondingly (up to ~306-494px). `center()`'s padding math doesn't know this, so the game list and state screens (loading/error/no-game/final/etc.) drift left with a large blank gap on the right whenever the cascade isn't on step A.

**Severity:** Cosmetic, not data-loss — but real, and it silently couples a layout constant in `display.ts` to a runtime value computed in `main.ts`. **Fix:** derive the line width from the active container width (pass it through, or compute it from `zoneGeometry`) rather than hardcoding it. This also becomes moot if §2.1's recommendation (fixed-width text zone) is adopted, since the cascade's variable-width layout goes away entirely.

### 3.2 Header line can overflow on long matchups

**Where:** `src/glasses/display.ts:54-58`, `renderPitchHeader()`

**Problem:** The second header line (`B: ${batterLastName} [${hand}]  P: ${pitcherLastName} [${hand}] ${pitchCount}p`) has no length cap. MLB has plenty of long/compound surnames (e.g. "Hernández", "Guerrero", doubleheader-roster call-ups). The HDR container is fixed at 36px tall (`main.ts:59`, `HEADER_H`); per the platform's text-wrap behavior, an overlong line wraps rather than truncates, which on a 2-line-budget container likely means the wrapped portion is clipped off-display rather than gracefully handled. **Fix:** truncate batter/pitcher last names to a max length with no ellipsis needed (space is tight) — verify the actual character budget on hardware/simulator first.

### 3.3 Raw SDK diagnostic strings are user-facing

**Where:** `src/glasses/zone.ts:344-362` (`formatZoneDiagnostic`), called from `main.ts` in `runCascade` and the `cascadeAllFailed` branch.

**Problem:** Independent of the §1.1 bug, `ZONE-DBG` output (raw SDK return strings like `sendFailed`, `imageException`, byte counts) is, by design, shown in the SPLITS column to *every* user during the first-pitch cascade probe, not just in a dev/debug build. This is appropriate for the team bringing the feature up, but isn't appropriate to ship — a fan at a ballgame doesn't benefit from seeing `B:80x96 1b / 1500chr imageException`. **Fix:** gate diagnostic-to-SPLITS behind the existing `?devReplay`-style dev flag, or simply behind a build flag; show a clean "loading zone…" or similar message to real users during the one-time probe instead.

### 3.4 Documentation describes three different versions of this feature

**Where:** `README.md`, `HANDOFF.md`, `docs/superpowers/specs/2026-05-23-zone-image-rendering-design.md`

**Problem:**
- `README.md` (latest commit) describes the image-cascade approach in broad strokes but doesn't mention the cascade, the 4 fallback steps, or that it's unconfirmed on hardware.
- `HANDOFF.md` (2026-05-13) describes the ASCII-grid era and concludes images are "not viable" — directly contradicted by the code that now exists, because it predates the 2026-05-23 reversal and was never updated.
- The 2026-05-23 design spec's own cascade table (3 steps, step C at 80×96) doesn't match the implementation (4 steps, step C at 40×48 — itself corrected in the *plan* doc but never back-ported to the *spec* doc).

**Fix:** pick one living document (recommend folding into `README.md` or a single `STATUS.md`) that reflects current reality, and retire or clearly date-stamp the others as historical record rather than active guidance. This is cheap and prevents the next session (human or agent) from re-deriving this same investigation.

## 4. UI/UX — concrete proposal if a text-zone redesign is taken (per §2.1)

This directly answers HANDOFF.md's open "Option A–F" question rather than leaving it unresolved:

- **Lead with the label, not the grid.** `Hi/L`-style two-axis labels are already implemented and tested — keep them, but spell out one axis in full and abbreviate the other so it reads at a glance without decoding (e.g. `High / In` instead of `Hi/L`, sized to fit the available width).
- **Show out-of-zone direction visually**, not just via label text (§2.2) — a single arrow glyph positioned just outside the grid edge closest to the actual pitch location communicates magnitude-free direction immediately.
- **Make the grid bigger relative to its column** than the original ASCII attempt — HANDOFF's Option D (use the full 120×144 zone-column height for a 5×5 or 7×7 grid) is worth prototyping since the *text* zone isn't constrained by the same BLE size economics as images; a denser grid costs nothing extra to send.
- **Keep the dot/empty distinction unambiguous** at whatever grid density is chosen — test on actual hardware/simulator with glow enabled, not just in a terminal.

## 5. Dead code / cleanup

### 5.1 Three exported render functions are unused in production

**Where:** `src/glasses/zone.ts`
- `renderZoneCanvas` (`zone.ts:324-329`) — comment says "kept; sendFailed on current SDK/hardware." Only referenced from `zone.test.ts`.
- `renderZoneCanvasRaw` (`zone.ts:335-340`) — produces a raw 4-bit pixel `number[]` directly (no PNG wrapping at all), with a comment noting it "matches the format used by flappy-g2 / demo-app-g2 which successfully call `updateImageRawData`." Never called from `main.ts`. Notably, this is a *much* simpler path than the PNG-encode-then-decode-to-bytes round trip `main.ts` actually uses (`renderZoneImage` → base64 PNG → `pngBytes()` decodes it back to raw bytes) — worth a quick experiment to see if sending this raw format directly is both simpler and more reliable, before investing further in the PNG cascade.
- `renderZoneBlank` (`zone.ts:319-321`) — only referenced from `zone.test.ts`.

**Fix:** if §2.1's recommendation is adopted (image cascade demoted to opt-in/experimental), keep these but mark them clearly as the experimental path's internals; if the cascade is dropped, delete these plus their now-unreachable tests (~150 lines across `zone.ts` + `zone.test.ts`, including the CRC32 table, Adler32, and `pngChunk` helpers that exist solely to support them).

### 5.2 Stale comment references a function that doesn't exist

**Where:** `src/glasses/zone.ts:85`
```ts
// encodePNG     — 8-bit greyscale (kept for reference; ~23 KB at 120×144)
```
No `encodePNG` function exists in the file — it was apparently deleted at some point and the comment wasn't. Harmless but confusing; remove the line.

### 5.3 Unnecessary network permission

**Where:** `app.json:16`
```json
"whitelist": ["https://statsapi.mlb.com", "https://react.dev"]
```
No source file references `react.dev` (confirmed via repo-wide search). This looks like a leftover from a template/scaffold. An app's declared network whitelist should be minimal — both for review/trust reasons and because it's the kind of thing a reviewer or a curious user will notice and ask about. **Fix:** remove it.

### 5.4 Repo hygiene — untracked working-tree clutter

**Where:** repo root and `.gitignore`

`git status` currently shows 34 untracked `glasses_<timestamp>.png` screenshots, an untracked `.superpowers/brainstorm/` directory (including `server.log`/`server.pid` process state from local tool sessions), and an untracked `src/dev/` (the dev-replay-fixtures module, which *is* real source imported by `main.ts:391` and should be tracked, not ignored). Also uncommitted modifications to `HANDOFF.md`, `app.json`, and `main.ts` sitting in the working tree.

**Fix:** add `glasses_*.png` and `.superpowers/` to `.gitignore`; commit or remove `src/dev/` deliberately; resolve the pending modifications before they're lost or compounded. (Flagging only — not resolving this without your go-ahead, since it touches what to commit.)

## 6. Performance / correctness

### 6.1 Possible duplicate polling when both glasses and phone UI are mounted

**Where:** `src/main.ts:382-405` (`main()`), `src/settings/SettingsApp.tsx:116-155`, `src/glasses/game-state.ts:200-203`

**Problem:** `main()` unconditionally calls `initSettingsPage()` (mounting the full React `SettingsApp`, including its own independent 10s `setInterval` poll of `fetchLiveFeed`/`fetchMatchupStats`), and then — if `hasFlutter` is true — *also* calls `startGlassesMode()`, which runs `game-state.ts`'s own independent 10s polling loop against the same endpoints. Per the Even Hub phone-side hosting model (the plugin runs inside a Flutter WebView), these two contexts may well be the same JS page on the same phone, meaning the app could be making **2× the network calls** it needs — same schedule, same live feed, same matchup stats, fetched and held in two separate state stores that never talk to each other.

**Fix:** verify with a network panel/log on an actual device whether both initializers run concurrently in production. If so, unify into a single poll feeding both the glasses bridge and the settings UI (or have one explicitly own polling and the other read from it via the existing `window` custom-event channel, which is already used for `strikezone:refresh`/`strikezone:sync`/`strikezone:game-changed`).

### 6.2 `displayInFlight` guard drops updates instead of queuing them

**Where:** `src/main.ts:250-260` (`refreshDisplay`)

**Problem:** The guard correctly prevents concurrent `updateImageRawData`/`textContainerUpgrade` calls (matching the platform's serialization requirement) — good practice, and worth keeping. But when a call is skipped because one is already in flight, the state change that triggered it is simply lost: there's no "render again after this one finishes" follow-up. Concretely: if a user taps to refresh while an auto-refresh-triggered render is still in flight, `state.atBat`/`state.game` get updated by `refresh()`, but the resulting `refreshDisplay()` call no-ops, and the user doesn't see their tap's result until the *next* unrelated render (next auto-refresh tick, or next gesture).

**Fix:** track a `pendingRender` flag set whenever a call is skipped due to the guard; in the `finally` block, check it and re-run once more if set.

### 6.3 Schedule-fetch failure is indistinguishable from "no games today"

**Where:** `src/glasses/game-state.ts:77-81` (`init()`)
```ts
try {
  state.games = await _fetchSchedule(todayDateString())
} catch {
  state.games = []
}
```
**Problem:** A network failure on the glasses-side schedule fetch is silently swallowed to an empty array, and `mode` is set to `'game-list'` regardless (`game-state.ts:86-90`). `renderGameList` with zero games shows only the header/footer lines with nothing in between — visually identical to "no games scheduled today." There's no error indication and no retry path (`onTap` in `game-list` mode does nothing when `games` is empty — `main.ts:343-345`). A user with a connectivity blip at app launch hits a dead-end screen with no way to recover except backing out and relaunching.

Contrast with `SettingsApp.tsx:106-109`, which correctly distinguishes this case (`setListError('Failed to load today's schedule.')`) on the phone side — the inconsistency is itself worth noting.

**Fix:** give the glasses-side `init()` the same distinction — a real `'error'` mode (already exists and is rendered, `display.ts:148-152`, complete with "Tap to retry") rather than collapsing to an empty game list.

### 6.4 No per-call timeout on network fetches

**Where:** `src/data/mlb-api.ts:153-175`

**Problem:** None of `fetchSchedule`/`fetchLiveFeed`/`fetchMatchupStats` use an `AbortController` or timeout. A hung network request leaves the app in `'loading'` indefinitely (no error surfaced, no retry offered) until the next 10s auto-refresh tick fires another overlapping request. Low-severity given the 10s polling cadence naturally retries, but worth a small, cheap fix (a few-second timeout per call) for snappier recovery and to avoid accumulating in-flight requests.

## 7. Test coverage gaps

- **`main.ts` has no tests at all** — it's the file with the actual orchestration bug (§1.1), the dynamic layout math (§3.1), and the cascade wiring. `zone.ts` and `display.ts` are well-unit-tested in isolation, but nothing exercises how `main.ts` calls them together, which is exactly where the §1.1 regression slipped through. Given the SDK bridge isn't mockable for free, at minimum the pure decision logic currently inlined in `main.ts` (e.g., "what does `renderPitch` send to INFO/SPLITS for each of: normal / contact / cascade-probing / cascade-failed") could be extracted into a testable pure function separate from the `bridge.*` calls.
- **`zone.test.ts` thoroughly tests three functions with no production callers** (`renderZoneCanvas`, `renderZoneCanvasRaw`, `renderZoneBlank` — see §5.1). Test coverage on dead code isn't harmful, but it's a signal worth noticing: the tests reflect what the team *intended* to wire up, which is a useful crumb trail, but also inflates the sense that this code is in active use.
- **`getDotPosition`'s out-of-zone band data exists but isn't visualized** (§2.2) — there's no test gap here exactly, but it's a reminder that the data needed for a better out-of-zone indicator is already computed and tested; only the rendering is missing.

## 8. What's working well (keep doing this)

- `displayInFlight` re-entrancy guard around the entire `_refreshDisplay()` — correctly serializes *all* bridge calls, not just image ones, matching the platform's own best-practice guidance.
- `matchupKey`/`matchupKeyRef` stale-response guards in both `game-state.ts:103-122` and `SettingsApp.tsx:130-138` — correctly prevent a slow `fetchMatchupStats` response for a previous at-bat from clobbering current state.
- `any` usage in `mlb-api.ts` is deliberately confined to the raw-API-parsing boundary (`parseScheduleResponse`/`parseLiveFeedResponse`/`parseMatchupStatsResponse`), keeping the rest of the codebase strictly typed — good isolation of an unavoidable any.
- The `?devReplay=1` dev-fixture path (`main.ts:390-396`, `src/dev/replay-fixtures.ts`) is a nice, low-cost way to exercise the full render pipeline offline, and is dynamically imported so it doesn't bloat the production bundle.
- `SCROLL_COOLDOWN_MS` debounce in `input.ts` for scroll events is a sensible guard against noisy hardware input.

---

## 9. Do NOT do

- **Do not retry the historical PNG-encoding-in-pure-JS approaches** already exhausted per `HANDOFF.md`'s "Image zone history" table (`canvas.toDataURL`, `CompressionStream`, raw `Uint8Array`/RGBA, base64-with-prefix) — all are documented dead ends specific to this WKWebView/SDK combination.
- **Do not just patch §1.1 in isolation and call the cascade done** — fixing the data-loss bug without addressing "has this ever worked on hardware" (§2.1) just means the *next* failure mode shows real pitch data next to a debug string nobody asked for, instead of nothing. Decide the strategy first.
- **Do not add a 5th cascade step at an even smaller size** if step D still fails in practice — per §2.1's legibility math, sub-50px images are already past the point of being useful; going smaller compounds the problem rather than solving it.
- **Do not silently resolve the uncommitted working-tree changes** (§5.4) as part of cleanup — those are pending edits to `HANDOFF.md`/`app.json`/`main.ts` that may represent in-progress work; surface them for an explicit decision rather than committing over or discarding them.
